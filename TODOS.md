## P1 ‚Äî Before First Real Seller (accepted 2026-05-21)

Critical gaps that surfaced in the review and were greenlit. All four shipped 2026-05-21.

- [x] **P1.1 ‚Äî OTP brute-force lockout** ‚Äî `src/lib/auth.ts` consumes the OTP code after 5 failed attempts (forces attacker to wait for a fresh code request, which won't come unless they're the real user). Simpler than a separate `lockedUntil` field; same brute-force resistance.
- [x] **P1.2 ‚Äî Rate limiting** ‚Äî `src/lib/ratelimit.ts` with ioredis sliding-window + in-process fallback. Wraps `/api/otp/send` (5/min/IP, 3/min/phone), `/api/cart` POST (30/min/user), `/api/reviews` POST (5/min/user), `/api/checkout` POST (5/min/user). 429 with Retry-After header.
- [x] **P1.3 ‚Äî Webhook signature verification** ‚Äî `/api/webhooks/stripe` uses `stripe.webhooks.constructEvent` against raw body; `/api/webhooks/razorpay` does constant-time HMAC-SHA256 verify. Both keep stub-mode for dev when secrets unset. Both reject `NO_SIGNATURE` / `BAD_SIGNATURE` with 401.
- [x] **P1.4 ‚Äî Refund dead-letter queue** ‚Äî New `payments.refund` BullMQ queue with exponential backoff (5 attempts). `disputes.resolveDispute` now calls `enqueueRefund` instead of `refundOrder`. Worker (`src/workers/ai.ts:startRefunds`) handles retries. Stuck refunds surface as `Order.status=REFUNDED` + `Payment.status=CAPTURED` mismatch (queryable signal for ops).
